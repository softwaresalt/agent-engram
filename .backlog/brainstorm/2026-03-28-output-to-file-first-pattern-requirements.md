# Output-to-File-First: Context-Efficient Agent Workflows

## Problem Frame

When agents perform research, analysis, or multi-step investigation, they
typically accumulate results directly in the context window: grep output, file
reads, analysis prose, intermediate conclusions. By the time the agent reaches
a decision point, the context window is dense with raw content that must be
re-read and reasoned over again. Sub-agents spawned with cheaper, smaller
models inherit a fraction of this context and must work with incomplete
information.

The fundamental issue is that the context window is being used as both a
working memory and a results accumulator. It is optimized for neither. A grep
invocation that returns 300 lines of raw matches to find one relevant function
costs 1,200+ tokens; the same query via `list_symbols` costs under 100. But
even engram tool responses accumulate when agents call many tools in sequence.

The "output-to-file-first" pattern addresses this by decoupling the *capture*
phase (agent writes its findings to files) from the *consumption* phase (agent
or sub-agent reads only what it needs via engram's search tools). Instead of
loading research into the context window, the agent writes structured
observations, findings, and decisions to `.engram/` files. Later, those files
are indexed into the document store and searchable via `query_memory`. Sub-agents
can find relevant findings with a single `query_memory` call rather than
inheriting a thousand tokens of raw output.

This brainstorm explores what infrastructure and tooling changes would make
this workflow a first-class pattern in engram.

## Requirements

### Observation Store

1. Engram MUST provide a dedicated `record_observation` MCP write tool that stores
   a structured agent observation to `.engram/observations/{branch}/observations.jsonl`.
2. An observation MUST contain:
   - `id`: UUID v4
   - `timestamp`: ISO 8601 UTC
   - `kind`: one of `research_finding`, `design_decision`, `implementation_note`,
     `bug_note`, `context_summary`, `tool_output_summary`
   - `topic`: short keyword topic tag (e.g. `"file-indexing"`, `"auth-flow"`)
   - `title`: one-line summary (≤ 120 chars)
   - `body`: structured text (Markdown or plain prose)
   - `source_tool`: name of the engram or CLI tool that produced the underlying data,
     if applicable (e.g. `"list_symbols"`, `"grep"`)
   - `source_refs`: list of file paths or symbol names referenced in this observation
   - `session_id`: optional Copilot CLI session identifier
   - `task_id`: optional backlog task ID
3. Observations MUST be appended to the JSONL file asynchronously (non-blocking)
   to avoid adding latency to the agent's workflow.

### Observation Indexing

4. After each `sync_workspace` or `flush_state`, observations MUST be ingested into
   the `ContentRecord` store (type = `"observation"`) so they are available via
   `query_memory` for vector search (when the `embeddings` feature is enabled).
5. When embeddings are disabled, observations MUST still be searchable via
   `list_observations` (keyword filter on `topic`, `kind`, and `title`).
6. A `list_observations` MCP read tool MUST return observations filtered by `topic`,
   `kind`, `since`, and `task_id`, with `limit` (default 20).
7. A `get_observation` MCP read tool MUST return a single observation by `id`.

### Context Summarization

8. A `summarize_context` MCP write tool MUST allow an agent to commit its current
   working context as a `context_summary` observation. The tool accepts a `title`
   and `body` parameter. The `body` is the agent's distilled summary of what it
   has learned so far in the current task.
9. Sub-agents MUST be able to use `query_memory("context_summary for {task_id}")` to
   retrieve the parent agent's committed summaries without inheriting the parent's
   raw context.

### Tool Output Summarization Pattern

10. Engram's tool response format MUST include, for high-token responses (above a
    configurable `summary_threshold_tokens`, default 500), a `_summary` field
    alongside the full response data. The summary is a 1-3 sentence distillation of
    the result generated by the daemon.
11. Agents operating under tight context budgets MAY request summary-only responses
    by passing `response_mode: "summary"` to any read tool. In this mode, only the
    `_summary` field is returned.
12. The `_summary` field generation MUST be non-blocking and best-effort: if summary
    generation fails, the full response is returned unchanged.

### Agent Instructions (Non-Tool)

13. The `AGENTS.md` file and Copilot agent instructions MUST be updated to document
    the output-to-file-first workflow pattern, including:
    - When to use `record_observation` instead of retaining results in context
    - How to use `query_memory` in sub-agents to retrieve parent agent findings
    - When `response_mode: "summary"` is appropriate (cheap model sub-agents)
    - The `summarize_context` pattern for handoffs between phases

### Dehydration and Lifecycle

14. Observations MUST be dehydrated to `.engram/observations/{branch}/` and survive
    daemon restarts.
15. Observations MUST be Git-trackable (not in `.gitignore`).
16. Observations older than 180 days SHOULD be pruned during `flush_state`.
    The threshold MUST be configurable via workspace config.

## Success Criteria

1. An agent performing a multi-file investigation calls `record_observation` to
   store findings. A sub-agent spawned for the next phase calls `query_memory` and
   retrieves the relevant finding in one call without needing the parent's context.
2. With `response_mode: "summary"`, a cheap sub-agent calling `map_code("some_fn")`
   receives a 50-token summary instead of a 2,000-token full call graph.
3. Observations written in one daemon session are retrievable after a daemon restart.
4. `list_observations(topic="auth-flow")` returns all auth-related observations for
   the current branch.
5. The `summarize_context` tool allows a planning phase agent to commit its context
   before handing off to a coding phase agent.

## Scope Boundaries

### In Scope

- `record_observation`, `list_observations`, `get_observation`, `summarize_context`
  MCP tools
- Observation JSONL model and per-branch storage
- Observation ingestion into `ContentRecord` (type = `"observation"`)
- `response_mode: "summary"` parameter on all read tools
- `_summary` field in high-token responses
- Dehydration and lifecycle management for observations
- Updated agent instructions in `AGENTS.md`

### Non-Goals

- Automatic context capture (agent must explicitly call `record_observation`)
- Full LLM-generated summaries in the daemon (the daemon is not an LLM client)
- Replacing `query_memory` with a new search primitive (observations are indexed
  as `ContentRecord` entries and searched via existing infrastructure)
- Real-time streaming of observation content to other connected agents
- Cross-workspace observation sharing
- Agent-to-agent direct messaging (observations are a one-way write → read pattern)

## Key Decisions

### D1: Explicit recording, not automatic capture

Agents decide when to record observations. Automatic capture (intercepting all
tool responses above a threshold) would produce noisy, redundant storage. The
pattern's value is in *curated* findings, not bulk logs.

### D2: Observations as ContentRecord type

Indexing observations into the `ContentRecord` store reuses the existing
`query_memory` infrastructure (which searches `ContentRecord` entries). No new
search primitive is needed. The `kind = "observation"` type filter differentiates
observations from specs, docs, and commit notes.

### D3: `response_mode: "summary"` as a client-side option

Rather than always computing summaries (expensive), summaries are opt-in per
request. This keeps the default path efficient and lets sub-agents with tight
context budgets opt in explicitly.

### D4: Daemon generates summaries without LLM

The `_summary` field MUST NOT require an LLM call in the daemon (that would
violate Constitution Principle VI on single-binary simplicity and add latency).
Instead, summaries are template-generated: "Found N symbols in M files, top
results: [names]." Simple heuristic summaries are far more useful than no
summary at all, and they are zero-cost.

## Outstanding Questions

### Resolve Before Planning

1. **Summary generation approach**: Template-based vs. extraction-based summaries?
   Template-based ("Found N results") is trivial to implement; extraction-based
   (extract the most relevant N lines from the response) is richer but more complex.
   Recommend template-based for initial release.

2. **`response_mode` parameter placement**: Should this be a top-level parameter on
   every MCP tool call, or a per-session configuration set via a `configure_session`
   tool? Per-call is more flexible; per-session reduces repetition for sub-agents
   that always want summaries.

3. **Observation visibility**: Should observations from a parent agent session be
   automatically visible to sub-agents (same workspace, same branch)? Yes — the
   JSONL file and `ContentRecord` entries are workspace-shared by design.

### Deferred to Implementation

4. **`source_refs` format**: Whether `source_refs` stores raw file paths, engram
   symbol IDs, or both. Symbol IDs enable forward-linking from observations to the
   code graph.

5. **`summary_threshold_tokens` default**: 500 tokens is a conservative starting
   point. The optimal threshold should be calibrated against typical response sizes
   for each tool.

6. **Compression for large observation bodies**: Whether long `body` fields (e.g.,
   a 5,000-character design decision) should be stored compressed in JSONL.
