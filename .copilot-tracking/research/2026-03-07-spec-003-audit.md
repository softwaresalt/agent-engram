## Gap List (Prioritized)

### üî¥ Critical Gaps (Spec-Committed Features Not Implemented)

#### GAP-001: Body Re-Derivation on JSONL Hydration (T080 ‚Äî marked ‚úÖ in tasks.md but NOT implemented)

**Spec commitment**: US7 acceptance scenario 2 states: *"When `set_workspace` hydrates the workspace, **Then** source files are parsed to populate node bodies"*. FR-163: after JSONL hydration, tree-sitter must re-parse source files to populate the `body` field on function/class/interface records.

**Actual state**: `hydrate_code_graph` in `src/services/hydration.rs:255` reads JSONL metadata and upserts nodes with `body: String::new()` (empty bodies). The JSONL does NOT contain source bodies (by design), and there is no post-hydration re-parse step. After a daemon restart, `get_active_context`, `map_code`, and `impact_analysis` will return **empty `body` fields** for all code symbols ‚Äî the primary value-delivery mechanism (FR-148: full source body guarantee) silently fails.

**Discovery source**: Phase 11 memory note: *"T080 (Body re-derivation after JSONL hydration) ‚Äî These were NOT implemented in this phase ‚Äî they remain for future work. The hydration.rs file was not modified."* Code confirms no tree-sitter parsing in `hydrate_code_graph`.

**Impact**: High. The entire FR-148 full-source-body contract breaks after any daemon restart. Affects `map_code`, `get_active_context`, and `impact_analysis` ‚Äî three of the seven core tools.

**Fix**: After JSONL hydration, for each hydrated code_file, read its source from disk and run `parse_rust_source()` to populate `body` fields on matched symbols (match by `body_hash`). Symbols whose `body_hash` doesn't match any parsed symbol need re-embedding.

---

#### GAP-002: Zero-Vector Null Serialization in JSONL (T107 ‚Äî marked ‚úÖ in tasks.md but NOT implemented)

**Spec commitment**: T107: *"Serialize `embedding` as `null` when all-zeros or embeddings feature disabled. Hydration: treat `null`/missing embedding as 'needs re-embedding'."*

**Actual state**: `dehydrate_code_graph` (line 661, 683, 705) serializes `embedding: f.embedding.clone()` unconditionally ‚Äî all-zero vectors (written when embedding model is unavailable) are serialized as `[0.0, 0.0, ...]` arrays. On hydration, these zero-vectors are loaded back as valid embeddings, causing silently degraded similarity search results.

**Discovery source**: Phase 11 memory: *"T107 (Zero-vector null serialization): Dehydration zero-vector handling not implemented ‚Äî dehydration.rs only stores body not embedding."*

**Impact**: Medium. Only affects deployments where embedding model was unavailable during indexing. Zero-vectors produce misleading cosine similarity scores.

**Fix**: In `dehydrate_code_graph`, check `if f.embedding.iter().all(|v| *v == 0.0)` and write `null` instead of the zero vector. In `hydrate_code_graph`, treat `null` embedding as `vec![]` (empty) and mark the symbol for re-embedding on next sync.

### üü° Known Gaps (Tracked, Not Blockers for Merge)

#### GAP-003: Atomic RELATE Upsert for `link_task_to_code` (T097 ‚Äî marked ‚úÖ in tasks.md but NOT atomically implemented)

**State**: Uses EXISTS-then-CREATE two-step (lines 1335‚Äì1347 in `write.rs`). A concurrent call between the EXISTS check and CREATE could create a duplicate concerns edge. Phase 11 memory explicitly notes: *"T097 not implemented separately; the existing EXISTS‚ÜíCREATE pattern remains (acceptable idempotency via check)."*

**Impact**: Low ‚Äî race window is very small; duplicate edge would be cleaned on next sync. Not a correctness blocker.

---

#### GAP-004: Unified Search ‚Äî Full Table Scan for Tasks/Specs (T093 ‚Äî partial)

**State**: Vector search for code symbols uses SurrealDB MTREE index (server-side). For context and spec tables, `unified_search` uses full-table loading + in-memory cosine similarity.

**Impact**: Low for typical workspace sizes. Becomes a performance issue at >10,000 context/spec entries.

## Recommendation

### Merge-Ready Assessment: ‚öÝÔ∏è Needs Work Before Merge

The branch is **functionally complete** for the happy path ‚Äî all 9 planned MCP tools are implemented with real logic, all build gates pass (262 tests), and 4 deferred items have proper ADR documentation. The codebase quality is excellent: no unsafe code, no unwraps in library code, comprehensive error propagation.

**However, two spec-committed features are marked complete in tasks.md but are not actually implemented:**

1. **GAP-001 (Body re-derivation)** is a user-visible correctness failure after any daemon restart. Any agent calling `map_code`, `get_active_context`, or `impact_analysis` after a restart receives empty `body` fields ‚Äî silently, with no error. This violates FR-148 (the primary value-delivery contract).

2. **GAP-002 (Zero-vector serialization)** is a data quality issue affecting embedding correctness after persistence round-trips.

### Recommended Action Before Merge

**Option A ‚Äî Implement GAP-001 + GAP-002** (recommended):
- Implement T080 in `src/services/hydration.rs`: after loading JSONL, walk each code_file path, call `parse_rust_source()`, match symbols by `body_hash`, populate `body` fields, flag unmatched as needing re-embed
- Implement T107 in `src/services/dehydration.rs`: serialize zero-vectors as `null`; update hydration to treat `null` as needs-re-embed
- Add integration test: `index_workspace ‚Üí flush_state ‚Üí clear DB ‚Üí set_workspace ‚Üí map_code` verifies non-empty body field

**Option B ‚Äî Document as Known Limitations + Defer** (faster):
- Write ADRs for T080 and T107 explicitly documenting the behavior after restart
- Update `get_active_context` / `map_code` / `impact_analysis` to return a warning field when `body` is empty (clients can trigger `sync_workspace` to repopulate)
- Mark tasks.md T080/T107 as deferred, not complete

Option A is strongly preferred ‚Äî the body-empty failure is invisible to callers and violates the core spec contract.
