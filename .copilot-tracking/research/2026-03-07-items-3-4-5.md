## Item 3: Dynamic Tool Proxy via IPC

### Current Architecture

**Shim side — `src/shim/transport.rs`:**

```rust
// lines 121–131
async fn list_tools(
    &self,
    _request: Option<PaginatedRequestParams>,
    _cx: RequestContext<RoleServer>,
) -> Result<ListToolsResult, ErrorData> {
    Ok(ListToolsResult {
        tools: crate::shim::tools_catalog::all_tools(),
        next_cursor: None,
        meta: None,
    })
}
```

`all_tools()` is a pure compile-time Vec<Tool> in `src/shim/tools_catalog.rs` — 700+ lines of Tool::new() calls with embedded JSON schemas.

**Shim side — `call_tool` (transport.rs:66-113):**

All tool calls already go over IPC:
```rust
let ipc_req = IpcRequest {
    jsonrpc: "2.0".to_owned(),
    id: Some(Value::Number(serde_json::Number::from(id))),
    method: request.name.to_string(),  // e.g. "create_task"
    params,
};
let response = crate::shim::ipc_client::send_request(&self.endpoint, &ipc_req, self.timeout).await ...
```

**IPC protocol — `src/daemon/protocol.rs`:**

Simple newline-delimited JSON-RPC 2.0. Three fields: `jsonrpc`, `id`, `method`, `params`. No method registry or introspection.

**Daemon dispatch — `src/daemon/ipc_server.rs:229-263`:**

```rust
match request.method.as_str() {
    "_health"   => { ... },
    "_shutdown" => { ... },
    method => match tools::dispatch(Arc::clone(state), method, request.params).await { ... }
}
```

**Daemon tools — `src/tools/mod.rs:33-97`:**

Pure match of 35 string literals → async function calls. No metadata (name/description/schema) stored anywhere in the daemon.

### Key Constraint: rmcp is shim-only

The `rmcp::model::Tool` type (which carries name, description, inputSchema) is **only imported in the shim**:

```
src/shim/tools_catalog.rs:14  use rmcp::model::Tool;
src/shim/transport.rs:12       use rmcp::model::{..., ListToolsResult, ...};
```

Searching `src/daemon/` for `rmcp` → **zero matches**. The daemon has no rmcp dependency. Adding `rmcp` to the daemon would require a Cargo.toml change and pulls in the full MCP SDK on the daemon side.

### Implementation Options

#### Option A: Add `_list_tools` to daemon returning raw JSON

The daemon handler returns the catalog as a `serde_json::Value` without using `rmcp::model::Tool`. The shim deserialises the JSON and wraps it in `ListToolsResult`.

**Changes required:**
1. `src/tools/mod.rs` — add a `pub fn catalog_as_json() -> Value` that encodes all 35 tool schemas as hardcoded JSON (duplicating the entire `tools_catalog.rs` content, but in the daemon crate).
2. `src/daemon/ipc_server.rs` — add `"_list_tools" => IpcResponse::success(id, tools::catalog_as_json())` arm.
3. `src/shim/transport.rs` — `list_tools` sends `_list_tools` IPC, parses response, builds `ListToolsResult`.
4. Sync risk: two copies of schema data (daemon JSON + shim `Tool` structs) drift independently — worse than before.

**Cost: HIGH. Benefit: NONE** (duplicates the problem).

#### Option B: Move catalog to a shared crate

Extract `tools_catalog.rs` into a new `engram-catalog` crate shared by both shim and daemon. The daemon depends on it to answer `_list_tools`; the shim depends on it to answer `tools/list` locally.

**Changes required:**
1. New Cargo workspace member `crates/engram-catalog/` with `rmcp` as a dependency.
2. Both `engram` (daemon) and `engram` (shim mode) depend on `engram-catalog`.
3. `src/daemon/ipc_server.rs` — add `_list_tools` arm calling `engram_catalog::all_tools()` serialized to JSON.
4. `src/shim/transport.rs` — either: (a) still call locally (same as today, but source is shared), or (b) call over IPC.

**Cost: MEDIUM–HIGH** (workspace restructuring). Eliminates sync risk if done correctly. But still adds rmcp as a daemon dependency transitively (since `Tool` type is rmcp).

#### Option C: Keep static catalog in shim (status quo + strengthen existing tests)

The shim serves `tools/list` from its own compile-time catalog. Sync is enforced by:
- `TOOL_COUNT = 35` constant (tools_catalog.rs:18)
- `tool_count_matches_dispatch` test (tools_catalog.rs:676–682) — fails CI if count differs
- Item 5's proposed exhaustive name-coverage test (closes the remaining gap)

The static catalog is the correct source of truth for schema metadata: descriptions and JSON schemas are documentation artifacts that belong with the shim (the MCP surface), not with the dispatch implementation.

**Cost: ZERO** (already done). Benefit: correctness maintained via compile-time checks.

### Recommendation: Defer — Option C is already correct

**The dynamic IPC proxy is not worth implementing now (or possibly ever).** Reasons:

1. **No runtime drift is possible today**: the shim crate owns both `tools_catalog.rs` (schemas) and calls `tools::dispatch` on the daemon. Any tool added to dispatch without a catalog entry fails the `TOOL_COUNT` assertion in CI.
2. **Schema metadata is presentation layer**: tool descriptions and JSON schemas are documentation for MCP clients, not implementation details of the dispatch. They belong in the shim (MCP surface), not the daemon (execution engine).
3. **Cost outweighs benefit**: adding `_list_tools` to the daemon either duplicates the catalog (Option A, worse) or requires workspace restructuring + rmcp dependency in daemon (Option B, medium cost for zero correctness gain).
4. **Item 5 closes the real gap**: the existing test only spot-checks 10/35 names. Exhaustive coverage (Item 5) eliminates the only remaining risk.

**If dynamic proxy is required in future** (e.g., for plugin-supplied tool schemas), Option B (shared catalog crate) is the right path. It should be driven by a concrete feature need, not speculative sync concerns.

### `ENGRAM_READY_TIMEOUT_MS` — Exact Behavior

Source: `src/shim/lifecycle.rs:28–46`

```rust
/// Default total wall-clock budget allowed for the ready-wait loop (milliseconds).
const DEFAULT_READY_TIMEOUT_MS: u64 = 10_000;

fn ready_timeout_ms() -> u64 {
    parse_timeout_ms(std::env::var("ENGRAM_READY_TIMEOUT_MS").ok().as_deref())
}

fn parse_timeout_ms(raw: Option<&str>) -> u64 {
    raw.and_then(|s| s.parse::<u64>().ok())
        .unwrap_or(DEFAULT_READY_TIMEOUT_MS)
}
```

- **Default**: `10000` ms (10 seconds)
- **Type**: `u64` milliseconds; non-numeric values silently fall back to default
- **Scope**: shim only; controls `poll_until_ready` in `ensure_daemon_running`
- **No CLI flag**: env-var only; not in `src/config/mod.rs`

### `ENGRAM_IDLE_TIMEOUT_MS` — Exact Behavior

Source: `src/daemon/mod.rs:122–132`

```rust
let idle_timeout = std::env::var("ENGRAM_IDLE_TIMEOUT_MS")
    .ok()
    .and_then(|s| s.parse::<u64>().ok())
    .map(|ms| {
        if ms == 0 { Duration::ZERO } else { Duration::from_millis(ms) }
    })
    .unwrap_or_else(|| plugin_config.idle_timeout());
```

- **Default**: falls through to `plugin_config.idle_timeout()` → `240 * 60 * 1000` ms = 14,400,000 ms (4 hours) from `src/models/config.rs:284` (`default_idle_timeout_minutes() = 240`)
- **Override**: `ENGRAM_IDLE_TIMEOUT_MS=0` means run forever (`Duration::ZERO`)
- **Scope**: daemon only; env-var takes priority over `.engram/config.toml` `idle_timeout_minutes`
- **No CLI flag**: env-var only; primarily used in test harness (`tests/helpers/mod.rs`)
