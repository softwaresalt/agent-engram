## Requirement Clarity

- [x] CHK013 — Is "highest-priority task" in `get_active_context` defined with a deterministic tiebreaker when multiple `in_progress` tasks share the same priority level? [Clarity, Spec §FR-127] — **Resolved**: Added clarification: tiebreaker is created_at (oldest first), consistent with get_ready_work in 002.
- [x] CHK014 — Is the scoring/ranking algorithm for `unified_search` specified beyond "merged, ranked results"? An agent cannot predict or debug result ordering without knowing the normalization strategy. [Clarity, Spec §FR-128] — **Resolved**: Added clarification: unified_search uses cosine similarity [0,1], code and task results ranked by raw cosine score.
- [x] CHK015 — Is the `depth` parameter in `impact_analysis` defined as counting code-only hops or code+concerns hops? An agent calculating blast radius must know whether concerns edges consume a depth unit. [Ambiguity, Spec §FR-129] — **Resolved**: Added clarification: depth counts code-only hops. Concerns edges are a single junction traversal (code→task), not counted toward depth.
- [x] CHK016 — Is the traversal algorithm (BFS vs DFS) for `map_code` and `impact_analysis` specified? Different algorithms produce different neighbor sets for the same depth. [Ambiguity, Spec §FR-126, §FR-129] — **Resolved**: Updated FR-126 to explicitly mandate BFS traversal; added clarification confirming BFS.
- [x] CHK017 — Is the `linked_by` field on `concerns` edges specified with an allowed format or value contract? An agent creating links needs to know what identifier to provide. [Clarity, data-model.md §concerns] — **Resolved**: Added clarification: linked_by is a free-form audit field (e.g. MCP client source_client). Updated data-model.md with the same note.
- [x] CHK018 — Is the `summary` field generation rule for Tier 1 nodes (`explicit_code`) explicitly defined? Data-model.md says "summary = body" — does an agent see the full raw source duplicated in the `summary` field? [Clarity, data-model.md §function, Spec §FR-143] — **Resolved**: Added clarification: For Tier 1, summary duplicates body for API consistency. Agents should use body for code content.
- [x] CHK019 — In FR-124 hash-resilient identity, is the behavior for renamed functions (different name, same `body_hash`) explicitly documented? The matching tuple is `(name, body_hash)` — a renamed function would not match, creating an orphaned `concerns` edge despite the body being unchanged. [Clarity, Spec §FR-124] — **Resolved**: Added clarification: renamed-function orphaning is intentional by design. Name+hash identity is the chosen tradeoff.
- [x] CHK020 — Is "4 chars per token" (FR-142) specified as a stable contract or an implementation heuristic? If an agent wants to predict tier classification for a symbol, it needs to know if this approximation is guaranteed. [Clarity, Spec §FR-142] — **Resolved**: Added clarification: 4 chars/token is a stable heuristic for tier classification, not a tokenizer contract.
- [x] CHK021 — Is the `symbol_name` input parameter defined to accept qualified names (e.g., `crate::billing::process_payment`) or only unqualified names (e.g., `process_payment`)? Ambiguity here causes agent link failures. [Clarity, mcp-tools.json §map_code, §link_task_to_code] — **Resolved**: Added clarification: symbol_name accepts unqualified names only. Qualified paths are not supported.
- [x] CHK022 — Is the relationship between the `max_traversal_nodes` config key (default 50) and the `max_nodes` tool parameter (default 50, maximum 200) clearly defined? Which takes precedence when they differ? [Ambiguity, Spec §FR-137 vs mcp-tools.json §map_code] — **Resolved**: Added FR-149 (config clamping: tool parameters silently clamped to config limits). Updated mcp-tools.json with effective_depth/effective_max_nodes output fields.

- [x] CHK045 — Is behavior defined when a source file is replaced by a directory with the same name (or vice versa) between syncs? [Edge Case, Gap] — **Resolved**: Added edge case: path type change (file→directory or reverse) treated as delete + skip. Directory is not a source file.

## Notes

- This checklist validates **requirements quality** (completeness, clarity, consistency, measurability) — not implementation correctness.
- Items reference spec sections as `[Spec §FR-NNN]`, data model as `[data-model.md §section]`, contracts as `[mcp-tools.json §tool]` and `[error-codes.md §code]`, and plan as `[plan.md §section]`.
- `[Gap]` markers indicate requirements that appear to be missing entirely.
- `[Ambiguity]` markers indicate requirements where an agent would need to guess or ask for clarification.
- `[Conflict]` markers indicate requirements that contradict each other across documents.
- `[Assumption]` markers indicate unstated dependencies on prior specs or external systems.
